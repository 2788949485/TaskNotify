# AGENTS.md

## 1. 项目概述

项目名称：**TaskNotify**

TaskNotify 是一个轻量级 Windows 桌面应用，用于自动检测长时间运行的开发任务、数据处理任务以及 AI 编程任务，并在任务完成、失败、超时或需要用户操作时发送 Windows 通知。

本项目最重要的产品要求是：

> 用户完成安装和一次性集成配置后，仍然按照原来的方式运行命令。普通使用场景不得要求用户在命令前添加 `tasknotify run`，也不得要求用户手动上报任务完成状态。

主要平台与技术栈：

- Windows 10、Windows 11
- C#、WPF
- .NET 10
- MVVM
- Windows App SDK App Notifications
- SQLite
- Named Pipe 本地进程间通信
- MVP 阶段使用 WMI 监控进程，后续可扩展 ETW

本文件规定所有人工开发者和 AI 编码代理在本仓库中必须遵守的规则。

---

## 2. 指令优先级

出现指令冲突时，按以下优先级处理：

1. 用户当前明确提出的需求
2. 本 `AGENTS.md`
3. 项目开发文档和架构文档
4. 已有自动化测试
5. 项目现有代码规范
6. 框架和语言的一般惯例

不得因为某种实现方式更简单，就擅自改变或弱化产品核心要求。

## 3. 核心产品约束

### 3.1 普通使用不得要求命令包装器

默认用户流程不得要求：

```powershell
tasknotify run -- python process.py
```

默认支持的用户流程是：

```powershell
python process.py
```

TaskNotify 应通过以下一种或多种方式自动检测任务：

- 官方 Hook
- PowerShell 集成
- VS Code 扩展事件
- 进程启动和退出事件
- 父子进程关系
- 可靠可用的退出码
- 输出文件规则
- 完成标记文件
- 最后兜底的活动状态推测

可以保留手动 CLI，用于诊断、高级集成或测试，但不得让其成为普通使用的必要条件。

### 3.2 进程退出不等于业务成功

进程消失只能证明该进程已经结束，不能证明业务执行成功。

通知文案必须遵守以下规则：

- 已可靠确认退出码为 `0`：显示“任务执行成功”
- 已可靠确认非零退出码或收到官方失败事件：显示“任务执行失败”
- 只知道进程结束，无法确认结果：显示“任务进程已结束，结果状态暂时无法确认”
- 仅依据 CPU、磁盘、文件稳定或窗口状态推测：显示“任务可能已经完成”

不得编写把未知进程退出直接标记为成功的逻辑。

### 3.3 每个任务必须包含两类可信指标

每个自动检测到的任务必须分别保存：

- `TaskProbability`：该进程有多大概率属于用户需要关注的任务
- `CompletionConfidence`：对任务完成结果的判断有多可靠

这两个概念不得混为一谈。

建议完成可信度从高到低为：

1. 官方集成或官方 Hook
2. IDE 或 Shell 生命周期事件
3. 可靠退出码
4. 完成标记文件
5. 进程退出事件
6. 输出文件稳定
7. CPU、磁盘或窗口状态推测

同一任务出现多个事件时，高可信事件覆盖低可信事件。

### 3.4 误报控制是一级功能

不得为了扩大检测范围而制造大量无意义通知。

默认模式为**平衡模式**，必须默认忽略：

- Windows 系统进程
- Windows 服务
- 浏览器子进程
- Electron 渲染进程
- IDE 语言服务器
- IDE Extension Host
- 调试适配器
- 自动更新进程
- 遥测进程
- 默认情况下的常驻开发服务器
- 未达到配置时长阈值的短任务

无法判断时，应先记录为候选任务而不通知，而不是贸然提醒用户。

## 4. 架构边界

解决方案应保持以下逻辑边界：

```text
TaskNotify.Desktop
TaskNotify.Core
TaskNotify.ProcessMonitor
TaskNotify.Integrations
TaskNotify.Ipc
TaskNotify.Infrastructure
TaskNotify.HookReceiver
TaskNotify.VsCodeExtension
```

### 4.1 `TaskNotify.Core`

允许包含：

- 领域模型
- 状态机
- 检测规则
- 可信度计算
- 接口定义
- 领域异常

不得依赖：

- WPF
- Windows App SDK
- SQLite 具体实现
- WMI
- VS Code API
- Claude 专用 JSON 模型

### 4.2 `TaskNotify.ProcessMonitor`

职责包括：

- 进程启动和退出事件
- 进程元数据采集
- 父进程和祖先进程解析
- 进程树聚合
- 活动状态采样
- 进程快照补偿与恢复

该层不得直接决定通知文案或 UI 展示，只能向核心层发送标准化检测事件。

### 4.3 `TaskNotify.Integrations`

每一种外部集成都必须将外部事件转换为 TaskNotify 的标准事件。

例如：

- Claude Code Hook → 标准任务事件
- PowerShell 命令生命周期 → 标准任务事件
- VS Code Task 事件 → 标准任务事件

集成专用 JSON、路径或 API 类型不得直接泄漏到核心领域模型中；必要时只允许作为可选元数据保存。

### 4.4 `TaskNotify.Desktop`

职责包括：

- WPF 界面
- 系统托盘
- Windows 通知展示
- 用户操作
- 设置页面
- 应用生命周期

不得在 WPF code-behind 中实现核心业务规则。

### 4.5 `TaskNotify.Infrastructure`

职责包括：

- SQLite
- Repository
- 日志
- 配置持久化
- 开机启动
- 文件系统抽象

基础设施层应实现 Core 或 Application 层定义的接口。

## 5. 任务状态规则

必须使用明确的状态机。不得允许任意服务直接修改任务状态属性。

支持的状态：

```csharp
public enum TaskState
{
    Candidate,
    Running,
    WaitingForInput,
    WaitingForPermission,
    PossiblyCompleted,
    Succeeded,
    Failed,
    Cancelled,
    TimedOut,
    EndedUnknown,
    Ignored
}
```

终态包括：

- `Succeeded`
- `Failed`
- `Cancelled`
- `TimedOut`
- `EndedUnknown`
- `Ignored`

必须遵守：

- 终态任务不得重新回到 `Running`。
- 后续低可信事件不得把 `Failed` 改为 `Succeeded`。
- 官方失败事件优先于推测完成事件。
- 重复终态事件必须具备幂等性。
- 任务重试应创建新的任务记录，并关联原任务；不得重新打开旧任务。

每增加一种状态流转，都必须增加对应单元测试。

## 6. 进程监控约束

### 6.1 MVP 监控方式

MVP 可使用：

- `ManagementEventWatcher`
- `Win32_ProcessStartTrace`
- `Win32_ProcessStopTrace`

定时进程快照只能用于补偿、恢复和校验，不得作为主要事件来源。

### 6.2 禁止无取消机制的永久轮询

所有监听器和轮询任务必须：

- 接收 `CancellationToken`
- 应用退出时能够正常停止
- 使用有上限的轮询间隔
- 不阻塞 UI 线程
- 能从临时 WMI 故障中恢复

不得添加没有取消机制和延迟控制的 `while (true)`。

### 6.3 进程元数据只能按尽力原则读取

读取命令行、可执行文件路径、进程所有者或父进程可能失败，例如：

- 进程退出过快
- 进程权限更高
- Windows 保护进程
- 进程属于其他登录会话

这些失败不得导致监控器崩溃。

应使用可空字段，并按合理日志等级记录。不得为了读取更多进程信息而让整个应用以管理员权限运行。

### 6.4 进程任务聚合

相关父子进程应尽量归并为一个逻辑任务。

示例：

```text
npm.cmd
└─ node.exe
   └─ tsc.exe
```

```text
python.exe
└─ ffmpeg.exe
```

属于同一用户操作时，不得为每个子进程单独弹出通知。

任务组只有在以下条件同时满足时才允许结束：

- 根进程已经结束
- 任务组中没有仍在运行的重要子进程

已知辅助进程可通过规则排除。

### 6.5 PID 重用

不得只使用 PID 作为持久身份。

进程身份至少应包含：

- PID
- 启动时间、事件时间戳或创建标记

避免 Windows 重用 PID 后把不同任务错误合并。

## 7. 检测规则约束

### 7.1 规则必须数据驱动

不得把所有检测逻辑写入一个巨大的 `if` 或 `switch`。

规则至少应支持：

- 进程名称匹配
- 可执行文件路径匹配
- 命令行匹配
- 父进程匹配
- 祖先进程匹配
- 最短运行时间
- 分数增减
- 通知策略
- 常驻服务分类
- 用户覆盖规则

内置规则和用户自定义规则必须能够区分。

### 7.2 正则表达式安全

支持正则时必须：

- 设置执行超时
- 保存前验证表达式
- 尽量避免灾难性回溯
- 不得无限制执行用户提供的表达式

### 7.3 用户规则优先

以下用户操作必须高于内置规则：

- 此类任务以后总是提醒
- 此可执行程序以后不再提醒
- 标记为常驻服务
- 标记为数据处理程序
- 修改最短通知时间

升级程序时不得覆盖用户规则。

## 8. 外部集成约束

### 8.1 Claude Code

Claude 集成必须：

- 优先使用官方 Hook
- 与现有 `settings.json` 合并
- 不得替换完整 `hooks` 对象
- 修改前创建备份
- 先写临时文件
- 验证 JSON 后再原子替换
- 卸载时只删除 TaskNotify 自己创建的内容
- TaskNotify Desktop 不可用时，Hook Receiver 仍应向 Claude 返回兼容成功结果
- 通知发送失败不得阻塞 Claude Code

默认隐私行为：

- 不保存完整 Prompt
- 不保存完整回答
- 不读取 transcript 文件
- 只保存事件类型、会话 ID 哈希、项目目录、时间戳和用户允许的摘要

### 8.2 PowerShell

PowerShell 集成必须：

- 作为独立模块安装
- 只在用户 Profile 中加入带标记的导入代码块
- 保留用户原有 Profile
- 保留用户自定义 Prompt
- 保留 Oh My Posh 和 Starship
- 保留 Conda 自动激活行为
- 保留 PSReadLine 快捷键和补全
- 不读取或保存完整历史文件
- 卸载时只删除自身标记块

必须使用以下标记格式：

```powershell
# TASKNOTIFY-BEGIN
Import-Module "..."
# TASKNOTIFY-END
```

不得覆盖用户的 PowerShell Profile。

### 8.3 CMD

默认实现不得修改全局 CMD `AutoRun` 注册表项。

CMD 第一阶段依赖进程监控和父子进程分析。

### 8.4 VS Code

VS Code 扩展应优先使用官方 Task 和 Shell Execution 事件。

必须：

- 使用 VS Code Task 生命周期 API
- Shell Integration 可用时才使用终端执行事件
- 通过本地 IPC 发送标准化事件
- 不读取终端屏幕内容
- 与进程监控去重

同一任务中，VS Code 官方事件优先于进程推测事件。

## 9. IPC 约束

除非经过明确架构变更批准，否则使用 Named Pipe 作为本地 IPC。

必须满足：

- 仅限本机
- 尽可能只允许当前用户访问
- UTF-8 JSON
- 明确的协议版本
- 单条消息最大 256 KB
- 使用长度前缀进行分帧
- 支持取消
- 支持连接超时
- 事件 ID 幂等
- 结构化错误响应

默认不得开放网络端口。

不得通过 IPC 提供任意远程命令执行能力。

Desktop 可以接收事件和查询请求，但不得执行通过 IPC 传入的不受信任命令。

---

## 10. 通知约束

### 10.1 普通用户权限

桌面通知进程必须以普通用户权限运行。

不得让整个应用默认要求管理员权限。

高权限任务应尽可能通过受控 IPC 将事件发送给普通权限的 TaskNotify 进程。

### 10.2 通知文案

通知文案必须准确反映完成可信度。

不得使用超过证据强度的表述。

### 10.3 通知操作

通知激活参数只允许包含安全的内部标识，例如：

- `taskId`
- `action`

不得在通知激活参数中嵌入未经验证的命令行或可执行文件路径。

打开路径前必须：

- 标准化路径
- 确认路径存在
- 确认该操作被允许
- 不自动执行 `.exe`、`.bat`、`.cmd`、`.ps1`

### 10.4 通知去重

同一个逻辑任务不得因为以下多个来源产生重复通知：

- WMI
- PowerShell
- VS Code
- Claude Hook
- 文件监控

关联键可根据可用信号组成，例如：

- 集成任务 ID
- 会话哈希
- PID 和启动时间
- 工作目录
- 命令哈希
- 启动时间窗口

结果优先级：

1. 官方失败事件
2. 官方完成事件
3. 可靠退出码
4. 进程退出
5. 文件推测
6. 活动状态推测

## 11. 隐私和数据处理

TaskNotify 必须坚持本地优先。

没有明确产品批准和用户主动选择时，不得增加：

- 云同步
- 遥测
- 用户行为分析
- 崩溃日志上传
- 远程日志

### 11.1 默认禁止采集的数据

- 源代码
- 数据文件内容
- 完整终端输出
- 完整命令历史
- Claude Prompt 或完整回答
- 浏览器内容
- 屏幕截图
- 剪贴板内容
- 键盘输入
- 密码
- API Key
- Access Token
- 数据库连接密钥

### 11.2 命令脱敏

保存命令摘要前，必须对以下敏感参数对应值进行脱敏：

- `password`
- `passwd`
- `token`
- `api-key`
- `apikey`
- `secret`
- `authorization`
- `connection-string`
- `database-url`
- `OPENAI_API_KEY`
- `ANTHROPIC_API_KEY`

脱敏必须在写入日志或 SQLite 之前完成。

不得先记录原始敏感命令，再事后脱敏。

### 11.3 日志

日志必须：

- 结构化
- 避免敏感载荷
- 使用滚动和保留期限
- 存放在当前用户目录
- 空闲状态下不持续写盘

对预期内的进程元数据读取失败，不应全部记录为 `Error`，除非确实影响核心功能。

---

## 12. 持久化约束

通过 Repository 或批准的持久化抽象使用 SQLite。

必须遵守：

- UI 和进程监控代码不得直接包含原始 SQL
- 数据库迁移必须有版本
- 迁移必须保证向前兼容和可恢复
- 用户创建的规则在升级后必须保留
- 数据库损坏不得阻止程序启动
- 备份和恢复文件必须保留在本地

默认保留期限：

- 任务历史：90 天
- 应用日志：14 天
- 任务完整输出：默认关闭

清理任务必须在后台运行并支持取消。

## 13. 编码规范

### 13.1 C#

必须：

- 启用 Nullable Reference Types
- 启用 Implicit Usings
- 异步 I/O 使用 `async`/`await`
- 长时间任务接收 `CancellationToken`
- 使用依赖注入
- 优先构造函数注入
- 合适场景使用不可变 record
- 架构边界使用明确接口
- 非 UI 通用库代码在合适情况下使用 `ConfigureAwait(false)`

避免：

- 全局可变状态
- Service Locator
- 任意静态服务访问
- 对异步调用使用 `.Result` 或 `.Wait()`
- 除事件处理器外使用 `async void`
- 在 WPF code-behind 中写业务逻辑
- 未观察异常的 fire-and-forget Task
- 捕获所有异常后直接吞掉

### 13.2 WPF 与 MVVM

- View 只负责展示
- ViewModel 不得直接访问 SQLite、WMI 或 Windows 通知 API
- 用户操作通过 Command 暴露
- UI 集合更新必须切回 Dispatcher
- 长时间操作不得运行在 UI 线程
- 用户关闭主窗口时默认最小化到托盘，只有选择“退出”才真正结束程序

### 13.3 命名

使用清晰的领域名称：

- `DetectedTask`
- `ProcessTaskGroup`
- `DetectionRule`
- `CompletionConfidence`
- `TaskProbabilityScore`

避免：

- `Manager2`
- `Helper`
- `CommonUtils`
- `DataObject`
- `TempService`

### 13.4 注释

注释应解释：

- 为什么存在某条检测规则
- 为什么需要某种 Windows 兼容处理
- 为什么某个低可信结果不能提升为成功

不要添加只是重复代码含义的注释。

公共 API 在含义不明显时应提供简洁 XML 文档。

## 14. 错误处理

### 14.1 通知失败不得影响用户任务

如果 Desktop、IPC、SQLite 或 Windows 通知失败：

- 能记录时记录错误
- 继续监控
- 不得终止或修改用户进程
- 不得改变外部程序退出码

### 14.2 集成接收器必须失败开放

Claude 和 Shell 集成接收器不得阻塞宿主程序。

事件无法发送时：

- 向宿主返回兼容成功结果
- 能记录时记录本地日志
- 不进行会延迟宿主进程的长期重试

### 14.3 恢复机制

WMI 或 ETW 中断后，进程监控器应自动恢复。

应用重启时：

- 对活动 PID 进行校验
- 只有身份可验证时才恢复任务
- 找不到对应进程的任务标记为 `EndedUnknown`
- 不得为停机期间遗漏的事件伪造成功通知

## 15. 性能约束

空闲目标：

- CPU 接近 0%
- 内存目标低于 150 MB
- 不持续写磁盘
- 不高频枚举全部进程

运行目标：

- 进程事件处理不得阻塞
- 规则匹配避免反复编译正则
- 高事件量时批量更新 UI
- 进程快照补偿间隔通常不得短于 30 秒

不得把全系统每秒轮询作为快捷实现。

核心行为发生变化时，必须同步增加或修改测试。

### 16.1 必须覆盖的单元测试

- 任务状态流转
- 可信度优先级
- 检测评分
- 内置排除规则
- 用户规则优先级
- 敏感命令脱敏
- 事件去重
- 进程组完成判断
- PID 重用
- 通知文案选择

### 16.2 必须覆盖的集成测试

- Named Pipe 分帧和大小限制
- IPC 并发客户端
- WMI 事件标准化
- SQLite 数据库迁移
- Claude 配置合并与卸载

### 16.3 必须覆盖的端到端场景

## 17. 完成定义

一个功能只有同时满足以下条件才算完成：

- 遵守架构边界
- 具有对应测试
- 没有在证据不足时提高通知可信度
- 没有无必要地要求管理员权限
- 没有新增敏感数据采集
- 支持取消和正常关闭
- TaskNotify 故障时不会阻塞外部工具
- 保留用户现有配置
- 必要时包含迁移或兼容处理
- 行为或配置变化后同步更新文档

功能在某一台开发机器上运行成功，不代表已经完成。

## 18. 禁止变更

没有用户明确批准时，任何开发者或 AI 编码代理不得：

- 让普通使用必须依赖 `tasknotify run`
- 把未知进程退出标记为成功
- 让桌面程序必须以管理员权限运行
- 禁用 Windows 安全控制
- 使用 OCR 读取终端屏幕文字
- 监听键盘输入
- 截取屏幕
- 持续读取剪贴板
- 上传任务历史到远程服务器
- 默认开启遥测
- 暴露公共 HTTP 服务或网络端口
- 通过 IPC 实现任意命令执行
- 覆盖 PowerShell Profile
- 覆盖 Claude `settings.json`
- 替换用户已有 Claude Hooks
- 默认修改全局 CMD AutoRun
- 保存完整 Claude Prompt 或回答
- 保存未脱敏的 Token、密码或连接字符串
- 添加无边界后台循环
- 添加高频全系统进程轮询
- 吞掉异常且不记录或恢复
- 删除测试来让构建通过

## 19. AI 编码代理工作流程

### 开始修改前

1. 阅读本文件。
2. 明确变更影响的架构层。
3. 检查已有接口和测试。
4. 判断是否影响可信度、隐私、去重或用户配置。
5. 优先选择最小且兼容的改动。

### 实现过程中

1. 控制改动范围。
2. 除非需求明确改变，否则保留现有用户行为。
3. 编写代码时同步添加或更新测试。
4. 将外部 API 和 Windows 进程元数据视为不可靠输入。
5. 所有宿主工具集成都必须失败开放。

### 完成前

1. 构建完整解决方案。
2. 运行相关单元测试和集成测试。
3. 确认没有覆盖用户配置。
4. 确认没有新增敏感日志。
5. 确认通知文案与结果可信度一致。
6. 总结修改文件、行为变化、测试结果和已知限制。

## 20. 初始开发优先级

除非用户明确要求其他顺序，否则按以下优先级开发：

## 21. 产品验收不变量

以下原则在任何版本中都必须成立：

1. 用户可以正常执行 `python script.py`，无需添加包装命令，也能收到通知。
2. 显示“执行成功”必须有可靠证据。
3. TaskNotify 自身故障不得改变用户任务的执行结果。
4. 必须保留用户现有 PowerShell、Claude Code 和 VS Code 配置。
5. 敏感命令值必须在持久化前完成脱敏。
6. 同一任务不得因多个检测来源重复通知。
7. 平衡模式应优先降低误报，而不是追求识别所有进程。
8. 所有长期后台监听都必须支持取消、异常恢复和正常退出。
